Scope: Architecture for the Parent application (frontend + backend) within the existing Mindforge monorepo, based on the Parents Portal PRD. Student and Teacher apps already exist; this defines parent-facing capabilities and backend changes without modifying Student/Teacher UX.

1. Goals & Non-Goals

1.1 Goals

  • Provide a dedicated Parent app (frontend + backend) with secure, read-only access to the parent’s own child’s academic progress, attendance, and fees.
  • Expose a Pay Info section (QR, UPI, bank details) as information only; no online payments.
  • Enforce strict isolation:
    • Parent accounts created only by Admin.
    • Each parent linked to exactly one student.
    • Each student has at most two parents.
    • Parent can only see data where student_id = linked_student_id.
  • Reuse Teacher/Admin services as sources of truth; Parent backend as a read-only façade.
  • Integrate cleanly into monorepo via apps/parent and the API Gateway.

1.2 Non-Goals (v1)

  • No write operations from Parent app (no editing data, no payments, no uploads).
  • No multi-student parent view in v1.
  • No teacher messaging or homework approval.
  • No redesign of Student/Teacher UX.

Key rules: Parent frontend → Gateway → Parent backend; Parent backend owns parent identity & linkage and exposes read-only aggregated views of academic, attendance, and fees for the single linked student.

3. High-Level Architecture

3.1 Components

Parent Frontend (React + Vite): Auth (mobile + MPIN), navigation (Dashboard, Academic Progress, Attendance, Fees, Pay Info, Profile), charts and lists per PRD.

Parent Backend (NestJS): Auth module, Parent profile & linkage, Academic Progress read, Attendance read, Fees read, Pay Info, Integration module, Parent DB + optional Redis.

3.2 Communication Patterns

Frontend → Gateway → Parent backend over HTTPS/JSON; Parent backend → Teacher/Attendance/Fees services via internal REST; Parent backend ↔ Parent DB/Redis via repositories.

4. Data Model & Ownership

Parent-owned tables: parent_accounts (mobile, mpin_hash, name, relationship, linked_student_id, status, audit fields) and login attempts (or Redis counters). Constraints: unique mobile; at most two parents per student (application-level rule).

Read-only domains: student master, tests/performance (Teacher/Test services), attendance (Teacher/Attendance), fees (Admin/Fees). Parent backend does not become a second source of truth.

5. API Contracts (Logical)

  • Auth: POST /parent/auth/login, GET /parent/auth/me.
  • Dashboard: GET /parent/dashboard (latest test, attendance %, fee summary).
  • Academic Progress: GET /parent/child/progress/tests, GET /parent/child/progress/summary.
  • Attendance: GET /parent/child/attendance/weekly, /monthly, /yearly.
  • Fees: GET /parent/child/fees/summary, optional /history.
  • Pay Info: GET /parent/child/fees/pay-info (QR + UPI + bank details, informational only).

6. Security, Roles & Data Sensitivity

Roles: Parent role only for this app; Admin/Teacher/Student interact via other apps/services.

AuthN: Mobile + 6-digit MPIN (hashed), rate-limited with lockout; JWT/session with role = parent and linkedStudentId.

AuthZ: Gateway enforces role; Parent backend ignores client-supplied student IDs and always uses linkedStudentId from token; upstream services treat Parent backend as machine client.

Data sensitivity: Student identity/performance, fees, and attendance treated as sensitive; HTTPS + encryption at rest.

7. Error Handling & Resilience

Standard error shape { code, message, details? } with appropriate HTTP status (400/401/403/404/429/500/503). Upstream failures mapped to domain-specific “temporarily unavailable” messages; frontend uses skeleton loaders, banners, and retry controls.

8. Key Risks & Mitigations

  • Data leakage across students: Always derive student ID from token; contract and security tests for isolation.
  • Weak MPIN-only auth: Rate limiting, lockout, strong hashing, optional future device protections.
  • Inconsistent data vs Teacher/Admin views: Read directly from source services; show “Last updated” where feasible.
  • Upstream downtime: Clear partial-outage messaging; short-term cached aggregates.
  • Misinterpreting Pay Info as payment gateway: Explicit UX copy; no “Pay now” actions.

9. Implementation Order (Architecture-Aligned)

Parent identity & linkage → Auth module → Profile read API → Integrations (Tests/Attendance/Fees) → Academic Progress APIs → Attendance APIs → Fees APIs → Pay Info config/API → Frontend flows (Auth, Dashboard, detail screens) → Observability.

Context summary (for UX)

New Parent app (React + Vite, NestJS backend) integrated via Gateway with role = parent. Parents are Admin-created, log in with mobile + 6-digit MPIN, see only one child’s data, and have read-only access to academic, attendance, and fee data plus Pay Info.

What UX SHOULD design now

  • Global navigation & layout for Parent persona: Dashboard, Academic Progress, Attendance, Fees, Pay Info, Profile.
  • Authentication flow: mobile + MPIN login, error/lockout/disabled states, session timeout behavior.
  • Dashboard screen: cards for latest test, attendance %, fee balance with clear links to detail pages.
  • Academic Progress screens: test list (with fields from PRD), subject/date filters, line & bar charts, empty/error/loading states.
  • Attendance screens: weekly/monthly/yearly views with % and lists/charts, visually consistent with Student attendance but tuned for parents.
  • Fees & Pay Info: fees summary + optional history; Pay Info with QR, UPI, bank details and explicit “informational only” messaging.
  • Profile: parent identity, mobile, relationship, linked child info, plus “Contact school/support” CTAs.

UX constraints & rules

  • Parent app is view-only: no edit affordances on academic, attendance, or fees; no payment actions.
  • Data must always be scoped to the single linked child; avoid multi-child UI in v1.
  • MPIN UX: numeric keypad, 6-digit entry, clear messaging for incorrect attempts and lockouts.
  • Visual language: align with Mindforge brand and Student/Teacher apps; prioritize clarity and quick scanning.

Open questions for UX / Product

  • Required historical depth for graphs (e.g., last 3 months vs full academic year).
  • How to visually communicate partial data availability (some domains temporarily unavailable).
  • Mobile vs desktop layout variations (if any) for v1.
