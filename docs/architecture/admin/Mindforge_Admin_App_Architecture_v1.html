Scope: Architecture for the Admin application (frontend + backend) within the Mindforge monorepo. Admin is the control plane for user approval, account lifecycle, and fee/payment configuration used by Student, Teacher, and Parent apps.

1. Goals & Non-Goals

1.1 Goals

  • Provide a dedicated Admin app to:
    • Approve/block student, teacher, and parent accounts, with default MPIN 000000 on approval and mandatory first-login MPIN change.
    • Configure and update payment details (QR, UPI, bank account) for the Parent app.
    • Define and modify annual grade fees (8/9/10) and extra subject fees per student.
    • Record and correct fee payments, with automatic reflection in Parent app.
    • Edit, activate/deactivate, and soft-delete user accounts with full audit trail.

1.2 Non-Goals (v1)

  • No classroom content management (Teacher domain).
  • No Student/Parent/Teacher UX redesign.
  • No payment gateway integration (Admin only records payments and configures static payment info).
  • No multi-school/tenant admin in v1.

Admin frontend → Gateway → Admin backend. Admin backend owns approval state, MPIN flags, fee configuration, fee ledger, payment info, and audit logs; other services call Admin for status/fee/payment info.

3. High-Level Architecture

3.1 Components

Admin Frontend: Sections for Users (approval & account management), Fees (grade & extra subject), Payments (entry & history), Payment Info (QR/UPI/bank), and basic Audit/Settings.

Admin Backend: User Management & Approval, Fee Configuration, Payment Entry & Ledger, Payment Info Configuration, Audit Trail, and Integration module; backed by PostgreSQL and optional Redis.

3.2 Communication Patterns

Admin UI calls Admin backend via Gateway (HTTPS/JSON, JWT with role = admin); Admin backend exposes internal read APIs for Student/Teacher/Parent services to check approval, MPIN flags, fees, and payment info.

4. Data Model (High-Level)

User & Approval: user_accounts with role, mobile, status, must_change_mpin_on_first_login, etc., plus parent/student link table.

Fees configuration: grade_fee_configs (per grade/year) and extra_subject_fee_configs (per student/subject).

Fee payments & summaries: fee_payments (per transaction) and optional student_fee_summary (denormalised per student/year).

Payment info config: payment_info_config with QR URL, UPI, and bank details.

Audit: admin_audit_logs capturing who changed what, where, and when.

5. API Contracts (Logical)

  • User approval & account management: list pending users; approve/reject; edit; activate/deactivate; soft delete; internal status lookup for other services.
  • Fees & payments: CRUD grade fee configs; manage extra subject fees; record/edit payments; compute per-student summaries; internal summary/payment-info APIs for Parent backend.
  • Payment info: Admin UI APIs to get/update QR/UPI/bank details; internal read for Parent backend.

6. Security & Compliance

AuthN: Dedicated admin login with JWT and role = admin. AuthZ: Gateway enforces role; Admin backend may extend with finer permissions later.

Data sensitivity: Minors’ PII, fee/payment data, and bank details treated as sensitive; HTTPS + encryption at rest + strict DB access. Audit: all changes to users, fees, payments, and payment info logged.

7. Error Handling & Consistency

Standard error shape { code, message, details? }, with 400/401/403/404/409/500 codes; grade/extras/summary updates are transactional or clearly labelled with last-updated timestamps; downstream apps treat Admin as source of truth for fees and payment info.

8. Key Risks & Mitigations

  • Incorrect fee configuration: strong context and confirmations; audit trail.
  • Misrecorded payments: edit + audit; clear separation of configured vs paid.
  • Unapproved/disabled users logging in: login flows call Admin status APIs and enforce status = ACTIVE and approved.
  • Admin misuse/accidental deletes: soft delete, confirmations.
  • Stale payment info in Parent app: Parent always reads from Admin; optional short cache.

9. Implementation Order (Architecture-Aligned)

Admin auth → user approval & MPIN flags → grade & extra fee config → payment entry & summaries → payment info config → internal service APIs → Admin UI screens → observability & audit views.

Context summary (for UX)

Admin app is the control plane: approves/blocks users, configures grade/extra fees, records/corrects payments, and manages payment info that drives Parent app views. Admin operations directly affect login access and fee data shown to parents.

What UX SHOULD design now

  • Navigation with sections for Dashboard, Users, Fees, Payments, Payment Info, Audit/Settings.
  • User approval flows with pending lists, detail views, approve/reject/activate/deactivate/delete actions, and clear messaging about default MPIN and first-login change.
  • Grade & extra fee configuration screens with strong context (grade, academic year, student).
  • Payment entry/history screens with per-student fee summary and editable payments (with notes).
  • Payment Info screen with QR upload, UPI/bank fields, and last-updated indicators.
  • Basic audit/log view filtered by entity.

UX constraints & rules

  • Admin actions are high-impact: destructive or wide-scope actions require confirmations.
  • Default MPIN is 000000; copy should clearly state that users must change it on first login.
  • Parent app remains view-only; all fee and payment configuration/editing is Admin-only.
  • Workflows should be efficient for handling many records (tables, filters, bulk patterns if added).

Open questions for UX / Product

  • Need for bulk approval/activation/deactivation in v1?
  • Required filters/exports for high-volume payment management?
  • Strength of confirmation patterns for destructive actions (typed confirms, etc.)?
