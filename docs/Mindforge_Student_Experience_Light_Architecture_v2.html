Project Name: Mindforge Student Experience
Idea Source: docs/Mindforge_Student_Experience.pdf (Student experience definition for Class 6–12, ICSE/CBSE/State Board)

v2 updates: (1) 6-digit MPIN-only login model for students. (2) Explicit multi-platform support with Android app, iOS app, and desktop browser all kept in sync via a shared backend.

1. Platform constraints

Primary form factors

  • Android app: First-class native or hybrid app for Android phones (low- to mid-range devices, intermittent connectivity).
  • iOS app: Equivalent experience for iPhones, functionally on par with Android.
  • Desktop browser: Modern desktop browsers (Chrome/Edge/Safari) with responsive layouts.

All three platforms share a common backend and data model so that homework, quizzes, tests, progress, and gaps stay in sync across Android, iOS, and desktop. Students can start on one device and continue on another with no manual migration.

Access pattern

  • Individual student accounts with login using a 6-digit MPIN only.
  • Parent/guardian and teacher/institute views remain out-of-scope for this iteration unless explicitly added later.

Performance & connectivity

  • Optimize for low bandwidth, especially on mobile: small app bundles, efficient APIs, lazy-loaded heavy AI interactions.
  • Core navigation and progress views must remain usable on unstable 3G.
  • AI calls can be deferred/queued with clear UI states and retry options.

Time horizon / scale (light assumption)

  • Single-region (India-focused) initial deployment with a future path to multi-region.
  • No complex multi-region consistency design in this phase.

Environment

  • Mobile: Android and iOS apps (native or cross-platform), sharing UX principles but respecting platform norms.
  • Web: Responsive web app for desktop browsers.
  • No assumptions yet about deep OS integrations (push notifications, etc.) or full offline mode.

2. AI constraints

Pedagogical behavior (non-negotiable)

  • AI must not give direct answers by default.
  • Enforced interaction pattern:
    Hints → Approaches → Conceptual Explanation → Counter-Questions → (Only if needed) Worked solution outline
  • Applies to homework help, quiz/test feedback, doubt solving, and absentee/gap-bridge flows.

Source of truth for content

  • Syllabus-aligned content and question banks driven by:
    • Class teaching data via Google NotebookLM (daily summaries, key concepts, examples).
    • Structured syllabus metadata (Class → Subject → Chapter → Topic).
  • AI must be constrained to syllabus-aligned materials; no open-internet knowledge for examinable content.

Determinism & safety

  • Model configuration should prioritize accuracy, consistency, and syllabus adherence over creativity.
  • On uncertainty, AI must:
    • Explicitly signal uncertainty (e.g., “I’m not fully certain; here is my best attempt.”).
    • Flag items that may need teacher/content-admin review where applicable.

Experience threads (AI roles)

  • Homework Generator
    • Builds daily homework tasks from NotebookLM-derived teaching summaries.
    • Focuses on conceptual understanding and application.
  • Quiz & Test Engine
    • Generates daily quizzes and chapter-level tests.
    • Evaluates responses and explains wrong answers.
  • Doubt Solver / Concept Coach
    • Reactively answers student questions.
    • Strictly follows the progressive guidance model.
  • Gap-Bridge Tutor
    • Re-teaches missed topics.
    • Identifies weak areas and proposes remedial activities.

Cost-awareness (light assumption)

  • For high-frequency interactions (e.g., MCQ grading), use structured, lower-cost AI or deterministic logic.
  • Reserve heavier reasoning for doubt solving, conceptual explanations, and complex custom exams.

3. Data sensitivity

User profile data

  • Student identity (name, class, board, school, contact details) is PII.
  • Encrypt in transit; encryption at rest strongly recommended.
  • Parent/guardian data, if later added, is also sensitive.

Learning and performance data

  • Homework completion, quiz/test scores, question-level attempts, and doubt logs are educational performance records.
  • They drive personalization and progress tracking and must be protected from unauthorized access and misuse.
  • Should support deletion/anonymization upon request where feasible.

Content and teaching data

  • NotebookLM-derived teaching data may include teacher IP and student context.
  • Treat it as internal educational content; avoid raw, uncurated exposure.

Regulatory posture (light assumption)

  • India-focused deployment with minors as users.
  • Be conservative about cross-border transfers to AI providers.
  • Make consent flows explicit for:
    • Account creation and login.
    • Use of data for personalization.
    • Use of third-party AI services.
  • Avoid unnecessary PII in prompts; prefer pseudonymous IDs and minimal context.

4. Integration assumptions

NotebookLM integration (teaching data feed)

  • Receives daily per-class feeds from NotebookLM summarizing:
    • Topics covered.
    • Key points and explanations.
    • Example questions (when available).
  • Used to generate:
    • Daily homework.
    • Daily quizzes.
    • Chapter-level assessments and gap-bridge flows.
  • Treated as a read-only integration; exact API to be finalized later.

Authentication & identity (6-digit MPIN)

  • Students log in using a 6-digit numeric MPIN only.
  • The same MPIN is used across Android, iOS, and desktop to access the same account.
  • Account identity (e.g., phone number or similar identifier) is captured at registration but is not part of the daily login UX.
  • Because MPIN-only auth is weaker than passwords/MFA, strong protections are required:
    • Online brute-force protection (rate limiting, lockouts, backoff).
    • Optional device-level protections (e.g., OS biometrics as a convenience layer).

Roles

  • Primary role: student.
  • Future (not in current scope): parent and teacher.

Question bank & evaluation

  • Mindforge backend is the system of record for questions, responses, and grading outcomes.
  • AI providers act as stateless generators/evaluators.
  • Outputs are stored only after backend validation/sanitization when needed.

Other integrations

  • No assumptions in this phase about school ERPs, payment gateways, or SMS/notification gateways.
  • These will be treated separately if added later.

5. Risk flags

R1: AI hallucination vs. syllabus accuracy

  • Risk: AI may generate off-syllabus or incorrect content.
  • Impact: Confused students; exam misalignment; erosion of trust.
  • Direction: Strong grounding in NotebookLM and syllabus metadata; validation for syllabus alignment.

R2: Over-helping and short-circuiting learning

  • Risk: Students may quickly bypass effort and demand full answers.
  • Impact: Weak conceptual foundations and dependence on AI.
  • Direction: UX must enforce progressive guidance; full solutions gated behind effort and reflection.

R3: Evaluation fairness and trust

  • Risk: Inconsistent or opaque AI-based grading decisions.
  • Impact: Score disputes; loss of trust.
  • Direction: Deterministic scoring for objective questions; clear explanations for wrong answers.

R4: Data privacy and consent (minors)

  • Risk: Mishandling minors’ data or unclear third-party sharing.
  • Impact: Legal, ethical, and reputational issues.
  • Direction: Minimize third-party data exposure; clear consent and privacy controls.

R5: Dependence on NotebookLM feed quality

  • Risk: Noisy or incomplete daily summaries.
  • Impact: Misaligned homework/quizzes; frustration.
  • Direction: Fallback to standard syllabus banks; label content origin (today’s class vs. standard syllabus).

R6: Connectivity constraints and latency

  • Risk: Over-reliance on live AI in low-connectivity contexts.
  • Impact: Sluggish or unreliable experience; abandoned sessions.
  • Direction: Pre-generate/cache content where possible; robust retry and offline states across all platforms.

R7: MPIN-only authentication strength

  • Risk: A 6-digit MPIN is vulnerable to brute-force guessing.
  • Impact: Potential account takeover if protections are weak.
  • Direction: Strong rate limiting and lockout; consider optional secondary factors (device binding, OS biometrics); clear lockout and recovery flows in UX.

1. Context summary (for UX)

Student learning app for Classes 6–12 (ICSE/CBSE/State), structured as Class → Subject → Chapter → Topic. Runs on Android app, iOS app, and desktop browser, all backed by the same account and data model so student work and progress remain in sync.

AI behaves as a teacher-like guide using the pattern:

Hints → Approaches → Concepts → Counter-Questions → (Optional) Worked solution outline

Daily flows covered: homework, daily quizzes, chapter tests, student-generated exams, doubt solving, and absentee/gap-bridge.

2. Key UX constraints (delta vs v1)

  • Multi-platform parity and sync: Core flows (login, home, homework, quizzes, doubt solving) must exist and feel coherent across Android, iOS, and desktop with shared state.
  • 6-digit MPIN-only login: Design fast, student-friendly MPIN entry, error handling, lockout, and recovery experiences.
  • Low-bandwidth mobile-first design: Desktop is supported, but constraints are driven by low-end Android.
  • Learning-first guidance: Every AI interaction visually expresses the stepwise guidance model instead of one-tap direct answers.

3. What UX SHOULD design now (with new constraints)

  • Cross-platform login and onboarding with 6-digit MPIN creation, confirmation, and entry.
  • MPIN error, lockout, and “forgot MPIN” entry points (even if backend specifics are TBD).
  • Navigation and IA (Home/Today’s Plan, Syllabus Browser) consistent across Android, iOS, and desktop.
  • Core learning flows (homework, quizzes, tests, custom exams, doubt solving, gap-bridge) with synced progress indicators.
  • Connectivity and sync states (sync in progress, waiting to sync, offline with queued actions).

4. What UX MUST NOT design yet

  • Backend/API specs or implementation details.
  • Full teacher/admin dashboards or school/ERP integrations.
  • Payment/subscription flows or rich parent analytics.

5. Open questions UX should flag back

  • How to best communicate MPIN lockouts and recovery to students and parents.
  • How to balance “remember me” / auto-login across devices with account safety.
  • How to clearly visualize “synced” vs. “pending sync” across platforms.
